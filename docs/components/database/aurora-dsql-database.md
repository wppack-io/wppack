# Aurora DSQL Database Bridge

**パッケージ:** `wppack/aurora-dsql-database`
**名前空間:** `WPPack\Component\Database\Bridge\AuroraDSQL\`
**Category:** Data

Aurora DSQL (Distributed SQL) 接続 Bridge。PostgreSQL wire protocol を使用しますが、IAM 認証 (SigV4 presigned token) と DSQL 固有の DDL 制約を扱う専用ドライバ。

## インストール

```bash
composer require wppack/aurora-dsql-database
composer require async-aws/core   # IAM トークン自動生成用
```

## DSN 設定

```php
// wp-config.php

// IAM トークン自動生成（推奨、ambient AWS 資格情報を使用）
define('DATABASE_DSN', 'dsql://admin@abc123.dsql.us-east-1.on.aws/mydb');

// 事前生成トークンを埋め込み
define('DATABASE_DSN', 'dsql://admin:eyJhbGci...@abc123.dsql.us-east-1.on.aws/mydb');

// カスタムトークン寿命・OCC retry
define('DATABASE_DSN', 'dsql://admin@host/mydb?tokenDurationSecs=600&occMaxRetries=5');
```

リージョンはエンドポイント (`abc123.dsql.us-east-1.on.aws`) から自動抽出されます。

## 主要機能

### IAM トークン認証
- SigV4 presigned URL を `async-aws/core` で自動生成
- `admin` ユーザーは `DbConnectAdmin` アクション、それ以外は `DbConnect`
- デフォルト寿命 900 秒 (15 分)、期限の 120 秒前にローテーション
- トランザクション中はローテーションを抑止 (接続切断で silent abort するため)

### OCC (Optimistic Concurrency Control) リトライ
- SQLSTATE `40001` / `OC000` / `OC001` 検出で指数バックオフ + decorrelated jitter (100ms → 2× → 5s cap)
- `occMaxRetries` (デフォルト 3) で `transaction()` 全体を再実行
- 個別ステートメント単位の自動 retry は行わない (副作用の multiplicative 回避)

### SSL
- `sslmode=verify-full` 強制 (opt-out 不可)
- libpq 17+ では `sslnegotiation=direct` を自動付与

### search_path / schema
- `PostgreSQLDriver` から継承 (`?search_path=...` / `?schema=...`)
- トークン回転後の再接続でも `applySearchPath()` が実行されるので schema が維持される

## DSQL 固有のクエリ翻訳

`AuroraDSQLQueryTranslator` は `PostgreSQLQueryTranslator` をラップ。MySQL-first ポリシーで、wpdb イディオムは silently 変換し、DSQL がサポートしない intent は `UnsupportedFeatureException` で reject します。

### Silent rewrite (MySQL → DSQL)

| 入力 | 変換後 |
|------|--------|
| `TRUNCATE TABLE t` | `DELETE FROM "t"` + `pg_get_serial_sequence()` reset |
| `CREATE [UNIQUE] INDEX` | `CREATE [UNIQUE] INDEX ASYNC` (DSQL は async index のみ) |
| `SERIAL` / `BIGSERIAL` | `GENERATED BY DEFAULT AS IDENTITY` |
| `FOREIGN KEY` / `REFERENCES` | constraint drop (DSQL は FK 未対応、MyISAM と同じ挙動) |
| `ON UPDATE CURRENT_TIMESTAMP` | `DEFAULT CURRENT_TIMESTAMP` のみ保持 (アプリ側で `updated_at` SET 必須) |

### Hard reject (`UnsupportedFeatureException`)

silent fallback が意図を歪めるため reject:

- `CREATE TEMPORARY TABLE`
- `CREATE TRIGGER` / `CREATE FUNCTION` / `CREATE PROCEDURE`
- `ALTER TABLE ... DROP COLUMN`
- `ALTER TABLE ... ALTER COLUMN ...` (type / default / NOT NULL 変更)
- `ALTER TABLE ... MODIFY COLUMN` / `CHANGE COLUMN`

カラム変更 migration は application-orchestrated copy パターンで:

```sql
ALTER TABLE t RENAME TO t_bak;
CREATE TABLE t (...new shape...);
INSERT INTO t (...) SELECT ... FROM t_bak;
DROP TABLE t_bak;
```

DSQL は DDL と DML を同一トランザクション内に mix できないので、各ステップを別コネクションで実行する必要があります。
